# Workspace Draft Completion Rule

Version: 2026.1

Status: FOUNDATION

---

# Purpose

This document defines when the AFEOL Workspace Framework Draft is considered complete.

It exists to prevent future redesigns caused by implementation details.

Every future discussion, developer, or AI assistant must follow these rules.

---

# Core Philosophy

The Workspace Framework is **not** an application.

The Workspace Framework is **not** a UI library.

The Workspace Framework is a **Runtime Platform** implementing the Workspace Standard.

Applications implement the Workspace Standard.

The Workspace Runtime never adapts itself to individual applications.

Applications adapt to the Workspace Standard.

---

# Architecture Principle

```text
Application

↓

Workspace Contract

↓

Workspace Runtime

↓

Canvas
```

The communication is duplex.

Applications send valid Workspace definitions.

The Workspace validates, renders, and returns structured results.

---

# Responsibilities

## Workspace Runtime

Responsible for:

- validation
- rendering
- layout
- Dock management
- Canvas management
- dialog management
- event dispatching
- diagnostics generation

The Workspace Runtime never owns application state.

---

## Application

Responsible for:

- business logic
- user preferences
- workspace persistence
- project state
- developer diagnostics
- temporary error registry
- application history

---

# Stateless Runtime

The Workspace Runtime is always stateless.

It never permanently stores:

- projects
- users
- preferences
- layouts
- history
- diagnostics

Every Workspace instance is created entirely from the data supplied by the application.

---

# Validation

The Workspace Runtime validates every request.

If the request satisfies the Workspace Standard:

```text
PASS
```

Otherwise:

```text
FAIL
```

The Runtime never silently fixes invalid input.

---

# Diagnostics

Diagnostics are generated by the Workspace Runtime.

Diagnostics contain:

- code
- severity
- message
- details
- suggestion

Example:

```text
Code:
WSP-0204

Severity:
Error

Message:
Widget limit exceeded.

Details:
Group "Appearance" contains 9 widgets.

Maximum allowed is 6.

Suggestion:
Split the group into multiple semantic groups.
```

---

# Error History

The Workspace Runtime never stores diagnostics.

Applications may optionally maintain a temporary diagnostics registry during development.

Typical lifecycle:

```text
Validation FAIL

↓

Application stores diagnostic

↓

Developer fixes problem

↓

Validation PASS

↓

Application clears temporary diagnostic
```

The Runtime only reports the current validation result.

The application decides whether diagnostics are stored or discarded.

---

# Developer Diagnostics Modes

Applications may expose developer diagnostics.

Recommended modes:

```text
DISABLED

ERRORS

VERBOSE
```

The Workspace Runtime is unaware of these modes.

---

# Workspace Components

The Runtime recognizes only generic workspace concepts.

Examples:

- Workspace
- Toolbar
- Dock
- Canvas
- Dialog
- Notification

The Runtime never knows application-specific concepts.

Examples that belong only to applications:

- Website
- Driver
- Booking
- Patient
- Truck
- Invoice

---

# Dock

The Dock is a Workspace component.

It is never application-specific.

The Framework owns:

- Dock rendering
- Dock positioning
- Dock movement
- Dock persistence rules
- Dock constraints

Applications own only the Dock content.

---

# Floating Dock

The Dock may support multiple presentation modes.

Examples:

- Left
- Right
- Floating
- Hidden

The Framework manages these modes.

Applications only request supported behavior.

---

# Context

The Workspace follows the current context.

Typical flow:

```text
User selects object

↓

Workspace updates context

↓

Dock displays relevant actions
```

The application never manually rebuilds the Dock.

---

# Canvas

The Canvas is always the primary working area.

The Dock always follows the Canvas context.

Never the opposite.

---

# Overflow

The Dock has finite capacity.

Applications must respect Workspace limits.

Overflow is solved by semantic grouping.

Never by arbitrary splitting.

Correct:

```text
Project

Appearance

Publishing

Developer
```

Incorrect:

```text
Tab 1

Tab 2

Tab 3
```

without semantic meaning.

---

# Compatibility

Applications become compatible by implementing the Workspace Contract.

Technology is irrelevant.

Compatible applications may be written using any language or framework.

---

# Design Rule

The Workspace Runtime defines the rules.

Applications follow those rules.

The Runtime is never modified for individual applications.

---

# Draft Completion

The Workspace Draft is considered complete when the following principles are defined:

- Runtime philosophy
- responsibilities
- stateless architecture
- duplex communication
- validation model
- diagnostics model
- Dock philosophy
- Floating Dock concept
- Canvas philosophy
- context model
- semantic grouping
- compatibility model

Implementation details intentionally remain outside this document.
